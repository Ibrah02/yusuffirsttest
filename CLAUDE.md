# PeerColab agent rules

## 1. Five load-bearing rules

Each rule below has three labels: **Failure mode this prevents** (the agent behaviour to watch for, including default-tool gravity), **Model** (why the rule exists), **Imperative** (what to do). Pure imperatives sleep under framing shifts; the model is what survives.

### Rule 1 — Library overview FIRST, files second

**Failure mode this prevents:** When the user names files or asks to investigate code, agents reach for `Read` / `grep`. Per-call cost is near zero; the library-overview op feels heavyweight. Cheap-path bias wins.

**Model:** Source files do not carry the mental model. Domain layers do. `GetLibraryOverview` is faster than `git grep` for orientation: the layer description IS the model the file structure implements; reading source re-derives prose that already exists.

**Imperative:** Before reading source for orientation, run:

```
peercolab cli run CLI.Systems.Libraries.GetLibraryOverview \
  --system 828dc457-f0ad-4f56-b61a-2a1268106191 \
  --library 4f6c8473-4549-4d51-bbd3-d5b989f78697
```

Then drill the relevant domain layer with `DescribeDomainLayer`.

### Rule 2 — Search the model BEFORE grepping

**Failure mode this prevents:** When the agent needs to locate an entity (operation, value type, property, layer) by name or partial match, it reaches for `grep` / `Read` over the printed PathItems. Cheap-path bias wins, the agent lands in the text layer instead of the structured model, and cross-library references get missed.

**Model:** PathItems are printed text; the model is a structured graph — entity → path → references → descriptions across every library. `CLI.Systems.CLISearchSystem` walks **the active system's model** — operations / value types / properties / layers defined *in the system you're CLI'd into*. It is NOT a catalogue of CLI operations themselves (`CLI.Systems.*`, `CLI.Documentation.*`, ...); the CLI surface is its own thing. (For PeerColab the two overlap because PeerColab defines its own CLI, but that's a self-reference quirk.) Grep cannot match descriptions or resolve cross-library references.

Search returns *entities*. When `run` / `man` errors, **the error already names the fix** — `Required scope: --system <id> --model <id>`, `Operation X is model-scoped; dispatch via ...`, `OperationNotFound ... not found in any ... CLI usage package`, etc. Read it, retry with what it says. Don't invent wrappers, retry with UUIDs, or grep-fallback.

Scope hint from the op's CLI path: `CLI.Systems.Models.*` ops want `--model`, `CLI.Systems.Libraries.*` ops want `--library`, plain `CLI.Systems.*` and `CLI.Documentation.*` want only `--system` (or no scope). When in doubt, run `man` on the op first — the `Required scope` line is verbatim.

**Imperative:** Before grepping for an entity name, run:

```
peercolab cli run CLI.Systems.CLISearchSystem \
  --system 828dc457-f0ad-4f56-b61a-2a1268106191 \
  --input '{"query":"<name or fragment>"}'
```

Reserve grep for non-PeerColab code (frontend internals, handler bodies, raw codegen output).

### Rule 3 — Documentation is the SHARED mental model

**Failure mode this prevents:** Agents writing layer descriptions tend to write *for themselves*: file paths and dispatch loops, because that is what the AI wants to see. The PM and new contributor cannot enter; the "shared" model collapses to an AI-private notebook.

**Model:** Domain-layer descriptions serve THREE readers, not one. The depth tree IS the audience curve.

| Reader | Enters depth tree at | Reads for |
|---|---|---|
| Product owner / end user | Root — the library description | What this is, why we built it |
| Developer / new contributor | Mid-tree | Install, use, then how it works |
| AI assistant maintaining the code | Architecture leaves | Dispatch loops, invariants, file-level detail |

The **root is the library package description** (set via `UpdateLibraryVersionPackageDescription`), not a layer — top-level layers are its children. See the LayerStyleGuide deep-dive for the full treatment.

**Imperative:** When you author or revise a description, **WHY** and **WHAT for the PM** is just as important as **HOW for the AI**. Architecture layers are LEAVES, not roots. The audience curve is a *depth-tree* rule — Simple-shape rollouts (single layer) don't need it; one paragraph for the developer audience is enough.

Deep-dive: `peercolab cli man CLI.Documentation.LayerStyleGuide`.

### Rule 4 — Specs live as domain layers, not as files or console output

**Failure mode this prevents:** Two sides. (a) When the user agrees on direction and the next move is to plan, agents draft in chat or in an `.md` file; the planning evaporates when the session ends. (b) When the user is still exploring direction ("where do we go from here?", "how should we approach this?"), agents over-fire R4 and draft a full rollout before alignment. Both miss the calibrator: chat for exploration, layer for commit.

**Model:** Plans, retrospectives, ticket specs, and TODOs need to persist across sessions and surface to teammates. Layer trees do that natively; chat and stray files do not. The layer description IS the spec — but only once there *is* a spec. Direction-setting comes first; the spec records the decision after.

**Imperative:** Exploratory questions get a 2-3 sentence recommendation in chat with the main tradeoff, not a drafted layer. Once the user agrees on direction, draft every plan, retrospective, ticket spec, or TODO as a domain layer in a library. Never as a console-only response. Never as an `.md` file in `docs/`, the repo root, or anywhere else. If the right library is unclear (cross-library work, or the user has not named one), **ask which library to use** before creating. The target library's top-level `Features & TODOs` layer always exists (auto-shipped on library creation, see §5); add the rollout as a child of it.

### Rule 5 — Model abstractions as operations or event bundles, not just code

**Failure mode this prevents:** When the agent sees a product-significant abstraction worth a name — a client-side service, a third-party HTTP API wrapper, a dispatch handler, a producer-side event surface — its training-data prior reads "PeerColab operation = HTTP API endpoint" and it writes the abstraction as a plain TS class / C# service / ad-hoc emitter. External HTTP surface ends up in the model; the internal abstraction does not. The PM, the new contributor, and the next AI session cannot discover it via `CLISearchSystem` at all. The shared mental model becomes asymmetric and `GetLibraryOverview` quietly stops being load-bearing for whole categories of code.

**Model:** Operations (and event bundles for fire-and-observe surfaces) are a **contract layer for naming abstractions**, not RPC. Usage packages classify operations as **outbound** / **inbound** (via `CommunicationDirection` on regular library `UsagePackage`s) or **internal** (via the separate `InternalUsagePackage` resource family). Code is the *implementation*; the operation or event bundle is the *contract*. The asymmetric default is what happens when the agent skips the *internal* classification.

**Imperative — the litmus test:** Before writing a TS class / C# service / handler / wrapper / `emit(...)` call, ask one question: **is this abstraction significant enough to be lifted to the product level?** Would a PM ever name it in product documentation, planning, or release notes? If yes, model it as an operation (request/response) or event bundle (fire-and-observe) first, then implement. Default to ops / event bundles even when nothing crosses HTTP — the internal classification covers exactly this.

- **Carve-out — granularity stays product-significant.** One product-significant abstraction = one operation (or one event bundle), not one method. Private helpers, state transitions, parsing helpers, memoization caches stay as plain code.
- **Carve-out — consuming events is plain code.** Model the producer (event bundle); subscribers / handlers stay as plain code. The engine subscribes natively.

Deep-dive: `peercolab cli run CLI.Documentation.OperationsEverywhereDoctrine` — decision table for the three classifications, worked examples for outbound / inbound / internal / event-bundle, create-link-print-implement workflow, non-example inventory. Companion for contract shape: `CLI.Documentation.ResourceFirstModellingDoctrine`.

## 4. Authoring rules for domain layers

**Failure mode this prevents:** Agents reading a flat bullet list parse it as a checklist and apply each bullet equally. The load-bearing rule (audience matches depth) flattens to peer rank with bookkeeping bullets. Result: layer descriptions that are technically rule-compliant but miss the constraint.

**Model:** One load-bearing rule (audience matches depth), supporting rules that follow from it. The worked example and three smells below make the constraint concrete.

**Imperative — when authoring a layer:**

- **Lead with the mental-model boundary** the layer carves out, then the **why**, then the how.
- **Markdown structure**: headers per concern, tables for option comparisons, numbered lists for flows, bullets for tracked items. Paragraph prose is bridging text only, not whole layer bodies.
- **Audience matches depth**: root = the library package description (PM/end-user what & why), top-level layers = developer (install/use), leaves = AI/architecture.
- **Architecture is leaves, never roots.** If a root description starts with file paths or class names, re-frame it.
- **Length doctrine**: layers >600 words trigger `consider-drilling`. Split into sub-layers rather than padding.
- **Mutation review** (mandatory): before and after any mutation, run `peercolab cli run CLI.Systems.GetMutationDocContext --system <SystemId> --input '{"entityType":"<type>","entityId":"<id>"}'`. Re-read the descriptions it returns; if the contract meaning shifts, update them in the same edit.
- **Deep-link other entities**: when the description names another operation / value type / alias / resource / event bundle / library / domain layer by id, format as a markdown link to its place in PeerColab Builder (`https://app.peercolab.com/system/{systemId}/...` with a `modelPath` or `libraryPath` query string). Readers click straight in instead of grepping for the id. URL routes + `modelPath` / `libraryPath` formats are in `peercolab cli man CLI.Documentation.LayerStyleGuide`.

### Worked example

Same library, same code, two candidate root descriptions for a `Cart` subsystem. The contrast shows what "audience matches depth" looks like at the file level.

**Architecture at root** (the failure mode):

> "State management and UX for the shopping cart in the storefront. Persists across reloads via `localStorage` and is available to anonymous and signed-in shoppers. Sibling of Catalog."

A PM cannot enter here. The first sentence names an implementation choice (state management, localStorage), not what the cart is or why it exists.

**Product at root, architecture below** (the audience-correct shape):

> "The customer's cart. Holds items the shopper is considering before checkout. Built so anonymous browsers can fill a cart, sign in, and not lose their selection. Architecture lives in sub-layers (`State`, `Hydration`, `UI`)."

The PM enters at "what the cart is, why it exists". Implementation moves to sub-layers where developers and AI find it.

### Three smells to watch for

Three patterns commonly fail the audience model. Watch for them when authoring a layer; each is defined in the LayerStyleGuide canon and is what layer-write op smell trailers will fire on.

- **Architecture-prose-at-root.** Trigger: root layer (no parent) opens with file paths, class names, or implementation prose. Why it matters: the PM / end-user lands at the root and bounces. Fix: lead with what the product is and why it exists; move implementation prose to a sub-layer.
- **No-headers-when-long.** Trigger: description over 150 words with no markdown headers. Why it matters: PM and AI both navigate by scanning, neither reads paragraphs. Fix: split into headers per concern.
- **No-why.** Trigger: first 200 words have no "why" or motivation. Why it matters: behaviour without motivation reads as arbitrary; readers cannot tell which behaviours are load-bearing. Fix: add one or two sentences answering "why does this exist, what problem does it solve" before implementation.

Deep-dive: `peercolab cli man CLI.Documentation.LayerStyleGuide`.

## 5. Workflow doctrine

### The working loop

**Failure mode this prevents:** The rest of §5 teaches rollout *mechanics* (the Features & TODOs convention, the Simple / Multi-phase / Grouped shapes, propagate-on-ship). An agent reads these as a filing system for specs and never registers that its own current activity — drafting the spec, writing the code, refining the prose — *is* the inner loop of a rollout already in flight. It keeps treating rollouts as artifacts to author rather than as the loop it is executing.

**Model:** Work in PeerColab is two nested loops.

- **Outer loop** — the system grows one rollout at a time. Every unit of work (build in a library, document the system, ship an external deliverable) is a rollout, each starting from the ground truth the last left behind.
- **Inner loop** — a rollout is itself **plan → iterate → roll out**: state the intent; iterate and refine until it becomes code or content and passes QA (often several QA passes — aim for the *right outcome*, not just *done*); then roll out — propagate the docs / code, and for an external deliverable, deliver it *and* fold discoveries back into the documentation.

**Imperative:** Recognize and name which phase you are in, and treat **"nothing is done until it is rolled out"** as a working rule. This is the lens for every subsection below — Adoption, the two flows, and the Features & TODOs convention are all *how* you run the loop, not separate activities.

### Adoption (precursor for codebases not yet modelled)

**Failure mode this prevents:** Agents land on a codebase that has a PeerColab system shell but no libraries — or libraries with no domain layers — and try to apply Contract-first or UX-first directly. Both flows assume a documented model exists. Without one, the agent authors contracts or UI flows against speculation, then has to backfill the documentation later under time pressure.

**Model:** Adoption is one-time per codebase; Contract-first and UX-first are recurring per feature. Adoption's output (a documented model) is the input both other flows assume. Skipping it forces every later flow to compete with the documentation backfill.

**Imperative:** When `GetLibraryOverview` in §3 returns no libraries or no domain layers, drill `peercolab cli run CLI.Documentation.DocumentExistingSystemDoctrine` and follow the four backfill phases (topology discovery → conceptual framing → contracts → external IOs). Subsequent Contract-first or UX-first work operates against the documented model the doctrine produces.

### Two development flows

**Failure mode this prevents:** Agents default to contract-first because editing the model feels more rigorous: pin the contract, then implement. For UI features this stalls the user journey. Contracts ship before the flow exists, then often need to change once the user QAs the journey.

**Model:** Two flows because two failure modes. Contract-first protects backend-driven changes from drift. UX-first protects user-visible flows from "we shipped a contract nobody can use".

**Upstream — pick the actor and artifact first.** Both flows below are *software* flows. They presuppose `CLI.Documentation.SolutionExplorationDoctrine` already chose which actor you serve and that the artifact is software, not a report, deck, or pure doc. That doctrine is the first solution-side step once an intention is broken down (the §3 greenfield hook); UX-first / contract-first come after. A radically different actor group is the signal for a separate model (bounded context), not a new layer on an existing one.

**Imperative:**

| | Contract-first | UX-first (default for UI features) |
|---|---|---|
| When | Refining existing surface; back-end change driven by need | New features with end-user visible flow |
| First | Edit the model via `peercolab cli run …` | Print contracts to the frontend, mock operations via PeerColab interceptor |
| Last | `RunPathItemsPrint` to refresh local files | User QA gate → BuilderServer impl behind a config flip. **Mocks stay forever** so dev / QA / offline still work without the backend; deleting them at cutover is the most common UX-first failure mode |

UX-first deep-dive: `peercolab cli man CLI.Documentation.UxFirstWorkflow`.

### Engine API reference

**Failure mode this prevents:** Agents land on the implementation step (write C# interceptors, TS mock interceptors, the composition root) without knowing where the engine's type surface is documented. Default-tool gravity sends them to reflection tests, codegen output, or guessed signatures. The agent then spends rounds compiling-and-fixing instead of reading.

**Model:** The engine is one library per host language (`@peercolab/engine` on npm, `PeerColabEngine` on NuGet, `peercolab-engine` on PyPI) with the same surface in idiomatic casing. The reference is a single doc op covering: `Result<T>` and its constructors, `TransportSession` + `SessionBuilder`, interceptor registration (`builder.Intercept(op.Handle(...))` / `session.intercept(...)`), inspectors (`inspectRequest` / `inspectResponse`), `ctx.GetAttribute<T>` + path parameters + characters, `Metavalues` (pagination, character timestamps), and the dispatch flow.

**Imperative:** Before writing engine code by hand, introspecting types via reflection, or guessing method signatures from codegen output, read `peercolab cli run CLI.Documentation.PeerColabEngineDocumentation`. Reach for it whenever a new engine concept appears in your work — `Result`, `ctx`, `intercept`, `Metavalues` are not codegen output; they live in the engine package and are documented there.

### Double-IO testing pattern (services)

**Failure mode this prevents:** Agents land on a service repo and write tests against a single store impl (mock-only, or DB-only) or against JSON fixtures. Mock-only tests pass in CI and fail in production when real-DB semantics diverge. DB-only tests are too slow to run on every commit, so iteration collapses or the suite gets cut. Fixture files bypass the operation contract and let service code drift from what tests assert.

**Model:** Two halves that hold together. (1) Operations are the only test surface — setup, execution, and assertions all go through `Create*` / `Get*` / `Search*` ops; no JSON fixtures, no direct store access. (2) Every IO surface ships dual impls (mock + real, picked at boot via `AppConfig.<...>Mode`); the SAME test suite runs against BOTH via NUnit `[TestFixture(...)]` multi-instantiation. The dual-suite catches mock-vs-real divergence; six interceptor structure rules (validate-early, batch-reads, single-write, stores-as-layers, no-store-in-store-injection, no-transactions-in-signatures) prevent divergence at the seam.

**Imperative:** When shaping a new service or a new IO surface, ship mock + real impls behind a config switch. **Author the in-memory impl first** — parallels UX-first, lets you iterate the store abstraction shape before committing to migrations. Add `[TestFixture(StoreBackend.InMemory)]` + `[TestFixture(StoreBackend.PostgreSql)]` to every store test class. The persistent impl follows once the in-memory dual-suite is green, and is often the least time-consuming step of the whole implementation.

Deep-dive: `peercolab cli run CLI.Documentation.DoubleIoTestingDoctrine` — six interceptor rules with code, reference wiring (composition root, test container, dual-suite fixture), and performance patterns for scaling the real-DB half (`[CallerFilePath]` schema partitioning, cached schema context, tuned Postgres `max_connections`).

### Batch reads and writes via multi-segment `run` / `man`

**Failure mode this prevents:** Agent issues N processes for N CLI calls. Each pays daemon roundtrip + CLI startup; the "one process per call" gravity sleeps even though the CLI has been multi-call capable for a while.

**Model:** `peercolab cli` accepts repeated `run` and `man` segments sharing top-level scope. For batches too large for argv (≥10 ops, or long JSON inputs), `--multi-man-run-input <file>` loads the segment list from JSON. Add `--fail-fast` to halt on first non-zero exit — **not transactional**, already-applied writes are not rolled back.

**Imperative — argv form for small batches:**

```
peercolab cli --system <id> --library <id> \
  run CLI.Systems.Libraries.DescribeDomainLayer --input '"<id-a>"' \
  run CLI.Systems.Libraries.DescribeDomainLayer --input '"<id-b>"' \
  man CLI.Systems.GetDocumentationContext
```

**Imperative — file form for large batches or long inputs:** `peercolab cli --system <id> --library <id> --multi-man-run-input batch.json` where `batch.json` is:

```json
[
  { "verb": "run", "operationId": "CLI.Systems.Libraries.DescribeDomainLayer", "input": "<id-a>" },
  { "verb": "run", "operationId": "Libraries.Library.DomainLayers.UpdateDomainLayerDescription", "inputPath": "./bodies/auth-rewrite.json" },
  { "verb": "man", "operationId": "CLI.Systems.GetDocumentationContext" }
]
```

Per-entry fields: `verb` ("run"/"man"), `operationId`, `input` (inline JSON) or `inputPath` (file path — mutually exclusive), optional `library` / `model` overrides. Top-level scope and `--fail-fast` apply across the whole file batch.

### Preview before committing — `--dry-run` and GetWipImpact

`--dry-run` replays a batch in an isolated store and returns the batch delta. `CLI.Systems.GetWipImpact` returns the cumulative WIP-vs-base delta. Both surface cascades — a delete also shows owned children removed and dependent ops/properties retyped. `--dry-run` and `--fail-fast` MUST come **before** the first `run`/`man` keyword (otherwise rejected — silent-commit footgun).

```
peercolab cli --dry-run --system <id> --model <id> run <Op> --input '{...}'
peercolab cli --system <id> run CLI.Systems.GetWipImpact --input '{}'
```

Deep-dive: `peercolab cli run CLI.Documentation.DryRunWorkflow`.

### Wire console inspectors before any mock

> *Canonical example of the (failure-mode, model, imperative) shape used throughout this block.*

**Failure mode this prevents:** PeerColab operations are dispatched through the engine, not `fetch`. They **do not appear in the browser's Network tab.** Without inspectors, the user sees the UI move but nothing in DevTools, and assumes the app is not actually calling modelled operations. This is the single most common "is anything happening?" moment in fresh repos.

**Model:** Async overlap is the asymmetry. Every request is meaningful (you want to see *that* an op fired). Only failed responses are interesting; success responses interleave out of order with other ops' request logs and create more noise than signal.

**Imperative:** When standing up a new frontend (especially with mock interceptors), wire `inspectRequest` to log `ctx.operation.id` for every call, and `inspectResponse` to log **only failures**.

Engine deep-dive (full inspector boot pattern): `peercolab cli run CLI.Documentation.PeerColabEngineDocumentation` (§8c).

### Surface every spec as a domain layer (Features & TODOs convention)

**Failure mode this prevents:** Restated from §1 Rule 3 because it is the most-violated rule. The convention details below say *how* to surface a spec; the rule itself is in §1.

**Model:** Layer trees are the **Single source of truth** for specs. They give every contributor one CLI flow to author / read / update specs, with transactional updates and dependency-impact references on every mutation. A spec living in two places drifts; "always a layer" removes the choice.

**Imperative:** Every spec, plan, retrospective, or TODO lives as a **domain-layer description** in a library, not as a `.md` file and not as a console-only response. The layer description IS the spec. Drive every author / read / update through `peercolab cli` operations.

#### Top-level Features & TODOs layer

Every library has a top-level layer named `Features & TODOs` that holds the rollouts (active initiatives). It is **auto-shipped on library creation** with the canonical body baked in, so the agent never needs to bootstrap it; just add rollouts as children. Canonical default tree:

```
Features & TODOs                   ← auto-shipped on library creation
├── TODO                          ← system-wide cleanup tracker (optional)
├── <Simple feature> rollout       ← single layer; description IS the full spec
├── <Multi-phase> rollout          ← parent + Specification + TODO children
│   ├── Specification              ← stable narrative
│   └── TODO                       ← status, milestones, QA gate
└── <Grouped initiative> rollout   ← parent + per-ticket sub-layers
    ├── <Ticket A>
    └── <Ticket B>
```

Add a rollout as a direct child of the existing `Features & TODOs` layer via `Libraries.Library.DomainLayers.CreateDomainLayer`, passing the F&T layer's id as `parentId`. For Multi-phase, add `Specification` and `TODO` sub-layers via `AddSubLayers` (or pass the recursive `subLayers[]` tree on the create call for Grouped). Cross-library work: ask the user which library should host the spec; default is the library most affected.

#### Rollout shapes

| Shape | When | Layer structure | Example |
|---|---|---|---|
| **Simple** | Ships in one PR; no separate QA gate | Single layer; description IS the full spec | A one-PR bug fix, a small renderer tweak, the §3 step 6 discovery hook |
| **Multi-phase** | Has a QA gate; status churns separately from scope | Parent + `Specification` + `TODO` sub-layers | A UX-first rollout (client journey → user QA → BuilderServer); a multi-PR feature with a staged cutover |
| **Grouped** | Multiple related tickets from one source (retro, theme) | Parent + per-ticket sub-layers (use recursive `subLayers[]` in one call) | Tickets from a retro session; a recording followup with N items grouped by area |

**Simple is the default.** Promote to Multi-phase only after status starts churning separately from scope; promote to Grouped only when multiple related tickets emerge. Failure mode of over-promotion: drafting Multi-phase with `Specification` + `TODO` children before M1 ships means the milestones are speculation, not commitment — the PM cannot tell which are decisions vs scaffolding, and the agent burns the user's time on plan ceremony when the next concrete step was the real answer.

#### Layer-authoring operations

- `CreateDomainLayer` — top-level layer, rollout, or sub-layer. Accepts recursive `subLayers[]` for one-shot tree creation.
- `AddSubLayers` — adds children under an existing parent.
- `UpdateDomainLayerDescription` — replaces the description text.
- `UpdateDomainLayerName` — renames in place.
- `MoveDomainLayer` — reparents (e.g. promoting a sub-feature into its own rollout).
- `DeleteDomainLayer` — removes a layer. Rollouts delete on ship; documentation stays. **Before deleting a rollout, run `CLI.Systems.GetWipImpact`** to list impacted `(library, layer)`, update each, sibling-role sweep. Deep-dive: `CLI.Documentation.LayerStyleGuide`.

#### What does NOT belong here

Documentation about the system itself lives in the *documentation* layers — the top-level layers besides `Features & TODOs` (e.g. `Systems`, `Workspace Contexts`, `App shell`). These layers describe what the system *is*; `Features & TODOs` describes the work *changing* it. Reference docs (`CLAUDE.md`, READMEs, contributor guides) live wherever the tool reading them looks — not under `Features & TODOs`. Generated docs (codegen output, type definitions) are outside this convention entirely.

### Never hand-edit generated PathItems

Files under any path containing generated PathItems (often `src/peercolab/`, `src/TransportAbstraction/PeerColab/`, or similar) are generated. Drive every model / library / system change through `peercolab cli run` operations; re-print after with `RunPathItemsPrint --library <id>` (default to the cwd-mapped library; ask before printing into other repos).

### Operations, not new CLI subcommands

New PeerColab functionality ships as webapp-exposed operations callable via `peercolab cli run <op>` and documented via `peercolab cli man <op>`. Not as new bespoke `peercolab cli foo` subcommands. The CLI binary stays thin; the webapp ships continuously.

Generalised in §1 Rule 5 — operations and event bundles are the contract layer for all product-significant named abstractions on the produce side, including internal-classified usage packages. Consuming events stays plain code, and the rule does not push method-level granularity.
